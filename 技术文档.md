# AI 作业批改项目技术文档

## 1. 项目说明

本项目是一个基于需求文档生成的 AI 作业批改本地 MVP，目标是跑通“上传文件 -> 解析题目和作答 -> 自动批改/辅助批改 -> 学情分析 -> 导出报告”的完整链路。

当前版本采用前端 + 本地 Python 服务实现，不依赖 npm 包。结构化文件会在浏览器本地读取；图片、PDF、ZIP 会发送到本机 `server.py` 做 OCR 和解包，不会上传外网。答题卡场景会在 OCR 文本中解析学生姓名、题号和作答内容。

## 3. 产品能力对应关系

项目按照需求文档中的三层题型能力设计。

### 3.1 自动批改题

对应页面：

- 批改总览
- 答案映射
- 复核队列

支持题型：

- 选择题
- 判断题
- 填空题
- 基础计算题
- 单一答案计算题

当前 MVP 中的处理逻辑：

```text
教师上传参考答案/评分规则文件
    ↓
浏览器解析题目、答案和分值
    ↓
学生上传作业文件
    ↓
规则批改并生成反馈
```

图片/PDF 原件会通过本地 `server.py` 调用 macOS Vision OCR，抽取文本后再进行答案识别。ZIP 会先解包，再按文件名排序逐个解析图片、PDF 或文本文件。复杂版面仍需要教师复核。

OCR 会返回：

- `average_confidence`：平均置信度，用于页面展示 OCR 准确性。
- `low_confidence_count`：低置信文本行数量。
- `line_count`：总文本行数量。

如果配置 `OPENAI_API_KEY` 或 `LLM_API_KEY`，服务会优先调用兼容 Chat Completions 的云端模型结构化题目。对于 `PNG/JPG/JPEG/WEBP`，服务会先尝试把原图直接发给视觉模型；对于 PDF，会先渲染前几页为图片；对于 ZIP，会抽取前几张图片或 PDF 页。视觉模型会结合题干、公式、图形和手写答案分析；如果视觉模型不可用，再退回 OCR 文本解析。云端不可用时，服务会自动尝试本地 Ollama 模型；本地模型也不可用时，再退回本地规则解析。

性能优化遵循“准确率优先”：默认 `PDF_RENDER_SCALE=3.0`，保持原始渲染清晰度；`VISION_FIRST=true` 只是在视觉模型可用时跳过前置 OCR 等待，仍然把清晰图片交给视觉模型直接判断。第二步学生卷如果同时需要补题，会把题目结构和学生作答合并到一次视觉请求，避免同一批图片重复读图。

上传接口会根据 `target` 区分业务语义：

- `target=reference`：第一步参考答案/评分规则，输出 `questions`。
- `target=submission`：第二步学生作业，输出 `students`。单人或少量图片/PDF 会尝试学生答案视觉解析；大批量 ZIP 会跳过视觉模型，使用 OCR 分段、姓名识别、文件夹/文件名合并，避免 50 人答卷一次性视觉解析导致超时。

如果第一步参考答案只包含答案/解析而缺少题目，前端会在第二步上传时附带 `need_questions=true`。后端只从学生试卷中抽取一次题目结构并返回 `questions`，其余学生只解析 `answers`，避免同一套题重复抽取。

模型解析链路：

```text
OCR 文本
    ↓
图片/PDF/ZIP 图片页优先走视觉模型直接看图
    ↓ 失败
云端 Chat Completions 结构化
    ↓ 失败
本地 Ollama 结构化
    ↓ 失败
本地规则解析
```

模型配置加载顺序：

```text
.env
    ↓
model_config.json
    ↓
系统环境变量覆盖
```

默认配置文件是 `model_config.json`。真实密钥建议放在 `.env` 中，不建议写进代码。

云端模型解析 OCR 的超时配置在 `server.py` 的 `get_cloud_timeout()`，可用 `.env` 中的 `CLOUD_LLM_TIMEOUT` 覆盖，默认 `600` 秒。老师在 OCR 文本框中修正文本后，可以通过 `/api/structure-text` 重新调用云端优先、本地兜底的模型结构化链路。

本地模型默认配置：

```json
{
  "local_model": {
    "enabled": true,
    "provider": "ollama",
    "api_url": "http://127.0.0.1:11434/api/chat",
    "model": "qwen2.5vl:7b",
    "vision_enabled": true,
    "vision_model": "qwen2.5vl:7b"
  }
}
```

可用 `.env` 覆盖 `LOCAL_LLM_ENABLED`、`LOCAL_LLM_PROVIDER`、`LOCAL_LLM_API_URL`、`LOCAL_LLM_MODEL`、`LOCAL_VISION_ENABLED`、`LOCAL_VISION_MODEL`、`LOCAL_LLM_TIMEOUT`。`/api/llm-test` 会同时检测云端文本、本地文本和本地视觉模型，并用 `effective` 字段说明当前会使用 `cloud`、`local` 还是 `rule`。

图片/PDF 解析链路优先级为：云端视觉模型直接看图；云端视觉失败时调用本地 Ollama 视觉模型；两者都失败时才使用 macOS OCR 文本和文本大模型兜底。

### 3.2 主观题辅助批改

对应页面：

- 主观复核
- 学情分析

支持题型：

- 语文作文
- 语文阅读理解主观题
- 历史材料题
- 政治论述题
- 数学计算大题
- 数学证明题

当前 MVP 中的处理逻辑：

```text
浏览器识别题型
    ↓
匹配评分框架
    ↓
提取学生答案命中点和遗漏点
    ↓
生成建议分、依据和反馈
    ↓
教师查看复核点
```

主观题不做全自动最终定分。AI 的角色是帮助老师完成阅读、拆解、标注、评语生成和复核提示。

主观题辅助批改必须同时使用三类信息：

- `title/stem/material`：题目原文、材料和设问。
- `answer`：标准答案、参考答案或参考要点；没有时可以为空。
- `rubric/keywords`：评分点、采分点或关键词。
- `student_answer`：学生作答。

如果缺少题目，系统会把题目回应度标记为“无法判断”，降低置信度，并进入重点复核。如果有标准答案，系统会把 `answer` 拆成参考要点并与 `rubric/keywords` 一起用于命中和遗漏判断；如果没有标准答案，则只按题目要求和评分框架做低置信辅助评分。

当前 MVP 使用启发式方式判断题目回应度；接入大模型后，应让模型完整读取题目、标准答案/参考要点、评分点和学生答案。

### 3.3 人工兜底题

对应页面：

- 复核队列
- 主观复核中的“数学作图题”

示例题型：

- 作图题
- 实验操作题
- 口语表达题
- 开放项目题

当前 MVP 中的处理逻辑：

```text
AI 识别题型
    ↓
判断当前版本不支持可靠批改
    ↓
不自动给分
    ↓
进入教师人工处理
```

### 3.4 答题卡

答题卡作为学生作业的一种输入形态处理。

当前 MVP 支持：

- TXT/CSV/JSON 结构化答题卡。
- 图片/PDF/ZIP 答题卡 OCR。
- 一个 ZIP 中批量上传多个学生作业。
- 识别 `姓名：张三`、`学生：张三`、`1. A`、`2. 42` 等常见格式。
- 客观题提取为题号答案。
- 主观题保留题号后的文本，进入评分框架辅助批改。

当前 MVP 暂不做标准化 OMR 涂点模板定位；如果答题卡是纯涂黑选项，建议后续增加模板配置和涂点识别。

学生作业 ZIP 处理规则：

- 后端会对 ZIP 内每个文件单独 OCR/解析，并返回 `sections`。
- 前端会把每个 section 解析成一个学生作业。
- 如果 OCR 文本中存在 `姓名：xxx`，使用该姓名。
- 如果 ZIP 中是 `学生名/01.png`、`学生名/02.png` 这种结构，会使用顶层文件夹名作为学生名。
- 如果同一姓名或同一学生文件夹出现多页，会合并答案。
- 如果没有姓名，则使用 ZIP 内文件名作为学生名。

## 5. 数据模型设计

当前 MVP 从上传的 JSON、CSV、TXT、MD 文件中解析数据，并在浏览器内存中生成批改结果。真实系统建议拆成以下模型。

### 5.1 Assignment

```json
{
  "assignment_id": "math_cn_20260601",
  "title": "六年级综合练习",
  "class_ids": ["class_01", "class_02", "class_03"],
  "upload_files": ["exam.pdf"],
  "status": "grading"
}
```

### 5.2 Question

```json
{
  "question_id": "q001",
  "assignment_id": "math_cn_20260601",
  "display_no": "1",
  "subject": "语文",
  "type": "essay",
  "grading_mode": "assist",
  "framework_id": "cn_essay_default"
}
```

### 5.3 AnswerMapping

```json
{
  "question_id": "q001",
  "standard_answer": "B",
  "answer_source": "teacher_uploaded_exam",
  "source_region": [120, 980, 300, 80],
  "confidence": 0.97,
  "teacher_confirmed": true
}
```

自动批改题需要 `standard_answer`。主观题可以没有标准答案，但必须有 `framework_id`。

### 5.4 GradingFramework

```json
{
  "framework_id": "cn_essay_default",
```

## 6. 未来后端服务拆分建议

真实落地时，可以拆成以下服务。

### 6.1 文件解析服务

职责：

- 处理图片、PDF、Word。
- 输出页面、区域、文本、题号和坐标。

输入：

```text
试卷文件
学生作业文件
```

输出：

```text
OCR 文本
题目区域
答案区域
置信度
```

### 6.2 题目结构化服务

职责：

- 识别题型。
- 生成 `question_id`。
- 区分自动批改、辅助批改、人工兜底。

### 6.3 答案映射服务

职责：

- 从教师上传试卷中识别参考答案。
- 将答案绑定到对应题目。
- 生成教师确认页数据。

注意：

- 只使用教师上传文件中的参考答案。
- 不从外部题库或 AI 生成内容中获取正式答案。

### 6.4 AI 批改服务

职责：

- 自动批改题：按标准答案判分。
- 主观题：按评分框架生成辅助初评。
- 人工兜底题：识别并提示人工处理。

### 6.5 教师复核服务

职责：

- 保存教师确认、修改和驳回记录。
- 将教师最终分数写入正式结果。
- 为后续同类题评分框架提供参考。

### 6.6 学情分析服务

职责：

- 统计班级错题。
- 汇总主观题共性问题。
- 生成教学建议。
- 追踪学生订正情况。

## 7. 前端开发逻辑

当前前端采用简单的“数据驱动渲染”。

核心流程：

```text
上传参考答案/评分规则文件
    ↓
parseReference 解析题目
    ↓
上传学生作业文件
    ↓
parseStudents 解析学生作答
    ↓
runGrading 执行自动批改和辅助批改
    ↓
render* 函数渲染结果和学情分析
    ↓
downloadReport 导出 JSON 报告
```

关键函数：

| 函数 | 作用 |
| --- | --- |
| `parseReference` | 解析参考答案/评分规则文件 |
| `parseStudents` | 解析学生作业文件 |
| `parseAnswerCardText` | 解析答题卡 OCR 文本中的学生姓名、题号和作答 |
| `extractAnswerCardAnswers` | 从答题卡文本中提取 `q1=A` 这类答案 |
| `gradeAuto` | 自动批改客观题和基础计算题 |
| `gradeAssist` | 辅助批改作文、历史政治主观题、证明题 |
| `analyzeQuestionIntent` | 解析题目要求，例如分析、说明、证明、作文扣题方向 |
| `deriveStandardPoints` | 合并标准答案、评分点和关键词作为参考要点 |
| `evaluateQuestionResponse` | 根据题目和学生答案判断题目回应度 |
| `tokenizeMeaningful` | 从题目中抽取用于回应度判断的核心词 |
| `gradeManual` | 标记人工兜底题 |
| `runGrading` | 执行整份作业批改 |
| `renderResults` | 渲染单个学生批改结果 |
| `renderAnalytics` | 渲染班级分析和复核队列 |
| `downloadReport` | 导出批改报告 |
| `switchView` | 切换左侧导航对应页面 |
| `showToast` | 展示操作反馈 |

服务端关键函数：

| 函数 | 作用 |
| --- | --- |
| `run_vision_ocr` | 调用 macOS Vision OCR，返回文本和置信度 |
| `parse_ocr_questions` | 本地规则解析题号和答案 |
| `llm_extract_questions` | 可选调用大模型结构化 OCR 文本 |
| `normalize_llm_questions` | 归一化大模型输出 |
| `load_dotenv` | 启动时读取 `.env` |
| `load_model_config` | 启动时读取 `model_config.json` |
| `is_llm_configured` | 判断模型是否已配置 |
| `extract_zip` | 解包 ZIP 并逐文件 OCR/解析 |

## 10. 当前版本边界

- 当前项目是本地 MVP，不包含真实 AI 模型调用。
- 当前可以真实上传 JSON、CSV、TXT、MD 文件并执行批改。
- 当前可以上传 PNG、JPG、WEBP、PDF，并通过本机 macOS Vision OCR 提取文本。
- 当前可以上传 ZIP，服务会解包并按文件名排序逐页 OCR/解析。
- 学生作业 ZIP 会按文件拆成多个学生，并尽量按 OCR 姓名合并多页。
- 当前提供 OCR 文本编辑和重新解析能力，用于处理答题卡表格行列被 OCR 打乱的情况。
- 当前支持 OCR 后的答题卡文本解析，但纯 OMR 涂点卡还需要后续模板化增强。
- 当前 OCR 对复杂手写、低清图片和复杂版面仍可能识别不准。
- 当前教师复核结果不写入数据库。
- 结构化文件只在浏览器中读取；图片/PDF 会发送到本机服务解析，不上传外网。
